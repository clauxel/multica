# Multica SaaS 商业方案

## 1. 结论先说

如果我们基于 `multica-ai/multica` 来做产品，我建议把商业化路径收得更窄、更轻：

**只提供云端托管版本，只服务个人开发者和小团队。**

一句话定位：

**让个人开发者和小团队，用云端 AI Agent 完成更多研发任务。**

这意味着产品策略要非常明确：

- 不做私有化
- 不做客户自部署
- 不做复杂企业销售
- 不主打中大型组织治理
- 只做开箱即用的云端 SaaS

最适合的产品形态不是“企业级 AI 管理平台”，而是：

**面向个人和小团队的 AI Agent 工作台。**

## 2. 为什么仍然适合选 Multica

虽然 Multica 原生支持工作区、Runtime、daemon、自部署等更重的能力，但它依然适合作为这个轻量 SaaS 的底座，因为它已经具备很强的协作和任务骨架：

- Agent 可以像成员一样领取任务
- 有 Issue、任务板、评论、状态流转
- 有工作区和成员机制，适合小团队协作
- 有 Skill 机制，适合沉淀常用工作流
- 有较完整的前后端基础设施
- 有开源社区和产品叙事基础

对于我们这条商业化路径来说，最重要的不是把 Multica 全部能力都卖出去，而是只抽取最适合云端产品的那部分：

- 云端任务分配
- 云端 Agent 执行
- 云端任务追踪
- 小团队共享 Skill
- 简单协作与通知

也就是说，我们要把 Multica 从“可扩展的 Agent 管理底座”，收缩成“开箱即用的云端产品”。

## 3. 最适合做成什么 SaaS

### 推荐产品方向

推荐包装成：

**面向个人开发者和小团队的 AI 编码协作 SaaS**

它不应该强调复杂治理，而应该强调这几件事：

- 比单独用 IDE Agent 更有组织
- 比聊天式 AI 更适合持续做事
- 比自己拼工作流更省时间
- 比企业型平台更轻、更快、更便宜

### 核心承诺

对外可以这样讲：

**把零散的 AI 编码使用方式，升级成真正可持续的云端协作流程。**

### 最适合的使用场景

- 个人开发者把重复研发任务交给 Agent
- 独立开发者同时跑多个任务
- 小团队把修 Bug、补测试、写文档、整理需求交给 Agent
- 团队成员共享高质量 Skill 模板
- 用一个看板管理“人做的事”和“Agent 做的事”

## 4. 目标客户是谁

### 核心客户

- 独立开发者
- AI Native 个人开发者
- 2 到 10 人的小研发团队
- 小型外包或工作室团队
- 正在高频使用 Codex、Claude Code、OpenClaw 的开发者

### 用户画像

最有可能付费的人通常是：

- 自己就是开发者，同时也是决策者
- 不想折腾部署和基础设施
- 希望马上开始用，而不是先做系统集成
- 已经感受到 AI 工具有效，但效率还不稳定
- 愿意为“更顺手、更省时间、更可持续”付费

### 不做的客户

- 大企业
- 高度定制化客户
- 强依赖私有化部署的组织
- 需要复杂权限和审批体系的团队

## 5. 客户痛点

针对个人和小团队，真正成立的痛点是：

- AI 工具很好用，但使用方式太零散
- 一个人同时处理多个 Agent 任务很容易失控
- 小团队缺少统一的任务可视化
- prompt 很难复用，换人后效果不稳定
- 团队没有一套轻量的“AI 协作工作流”
- 不想自己部署、维护、调模型、接运行环境

所以这个产品最该卖的不是“强治理”，而是：

- 更顺手
- 更省时间
- 更适合持续使用
- 更适合多人轻协作

## 6. 商业模式建议

### 推荐模式

建议采用简单清晰的两层收入结构：

1. 订阅费
2. 使用量增购

也就是：

- 基础套餐按月或按年收费
- 超出配额后按 Agent 任务量、执行时长或高级额度增购

### 不要做重销售模式

既然只针对个人和小团队，就不建议把商业模式做成重销售：

- 不靠企业年框
- 不靠私有化项目
- 不靠高价实施服务

更适合的是产品自助增长：

- 7 天试用
- 低门槛付费
- 明确套餐边界
- 自助升级

### 核心收费逻辑

建议收费重点放在：

- 云端 Agent 使用额度
- 并发任务数
- 团队成员数
- 高级 Skill 或模板库
- 更高执行速度或优先队列

## 7. 定价方案建议

建议从三层开始，不要一上来做太复杂。

### Starter

作为最低门槛付费档，适合个人开发者先开始用。

- 价格：`10 美元 / 月`
- 建议包含：
  - 1 个工作区
  - 1 位成员
  - 基础月度任务额度
  - 基础任务板
  - 基础 Skill 使用
  - 1 个并发 Agent 任务

### Pro

适合个人开发者和独立开发者。

- 价格建议：`24-39 美元 / 月`
- 建议包含：
  - 更多月度任务额度
  - 更多并发任务数
  - 更完整的历史记录
  - 自定义 Skill
  - 更快的云端执行优先级

### Team

适合 2 到 10 人小团队。

- 价格建议：`79-199 美元 / 月`
- 建议包含：
  - 2 到 10 名成员
  - 团队共享工作区
  - 团队 Skill 库
  - 更高任务额度
  - 更高并发数
  - 基础成员管理

### 使用量增购

建议做成简单直观的附加包：

- 额外任务包
- 额外执行时长包
- 更高并发包

## 8. 产品包装建议

### 当前最适合卖的部分

基于现有 Multica，第一阶段最适合包装出来的价值是：

- 一个云端任务工作台
- 一个可以持续做事的 Agent 系统
- 一个适合个人和小团队的轻协作面板
- 一个可复用的 Skill 空间

### 商业化前建议优先补齐的能力

如果我们按“纯云端 SaaS”去做，最值得优先补的是：

- 订阅和计费
- 使用量统计和配额控制
- 新手引导
- 云端执行状态可视化
- 结果历史和任务检索
- 团队邀请和基础成员管理
- 稳定的失败重试和错误提示

### 不建议优先做的能力

在这条路线里，下面这些先不要做重：

- 私有化部署能力
- 企业级 SSO
- 复杂审批流
- 多层权限体系
- 高度定制化集成

## 9. SaaS 站点怎么搭

既然只服务个人和小团队，官网和产品站的目标就不是“打大客户”，而是：

**让用户看完就能马上注册试用。**

### 推荐站点结构

`/`
首页，核心价值、主要场景、截图、CTA

`/pricing`
价格页，清楚展示 Starter / Pro / Team

`/features`
功能页，展示任务板、Agent、Skill、协作方式

`/use-cases/solo-dev`
个人开发者场景页

`/use-cases/small-team`
小团队场景页

`/templates`
Skill 模板或预设工作流页

`/docs`
帮助文档

`/blog`
内容和案例页

`/login`
登录页

`/signup`
注册页

### 首页文案方向

首页标题建议：

**你的云端 AI 开发搭子，帮你持续推进任务。**

首页副标题建议：

**把修 Bug、补测试、写文档、做重构这些工作交给云端 Agent，在一个轻量工作台里持续推进。**

### 宣传语备选

下面这组文案可以直接用于后续 SaaS 站首页、功能页、价格页和广告素材。

#### 主标题型

- 你的云端 AI 开发搭子，帮你持续推进任务。
- 把零散的 AI 编码，变成持续推进的任务流。
- 不只是聊天，更是能交付的 AI 工作台。
- 一个人，也能同时推进多个研发任务。
- 小团队的第一套 AI Agent 工作台。
- 让 AI 不只回答问题，还能把事情做下去。

#### 价值表达型

- 注册就能用，不用自己部署。
- 把修 Bug、补测试、写文档交给云端 Agent。
- 让个人开发者也拥有自己的 Agent 工作台。
- 让小团队共享 Skill、共享进度、共享结果。
- 从 prompt 到任务流，把 AI 真正用进日常开发。
- 把会写代码的 AI，变成真正能协作的帮手。

#### 副标题型

- 在一个轻量工作台里创建任务、分配给 Agent、追踪进度，把重复研发工作持续推进。
- 适合个人开发者和 2 到 10 人小团队，用更省心的方式管理云端 AI Agent。
- 不用折腾部署，不用来回切聊天窗口，让 Agent 在云端持续做事。
- 从 Bug 修复到测试补全，从文档整理到重构执行，让云端 Agent 成为稳定生产力。

#### 短口号型

- 让 AI 持续做事。
- 让任务自己往前走。
- 少一点来回提示，多一点持续交付。
- 让开发效率真正跑起来。
- 你的任务，交给云端 Agent。
- 轻量协作，持续推进。

### 首页最该强调的 4 个卖点

- 不用部署，注册就能用
- 一个账号就能同时推进多个任务
- 个人也能有“Agent 工作台”
- 小团队可以共享 Skill 和进度

### CTA 建议

首页主按钮建议：

- 开始试用
- 立即试用

次按钮建议：

- 查看演示
- 查看价格

## 10. 市场进入策略

### 适合的获客方式

既然只打个人和小团队，最适合的是 PLG 和内容驱动：

- GitHub 开源流量导流
- X、Reddit、Product Hunt、开发者社区
- 独立开发者内容渠道
- YouTube / B 站产品演示
- SEO 内容
- 模板和使用案例传播

### 不建议的获客方式

- 长销售周期的大客户拜访
- 重线下商务拓展
- 复杂招投标

### 最强切入话术

建议对外强调：

**如果你已经在用 AI 写代码，现在你需要的不是更多聊天窗口，而是一个能持续推进任务的云端工作台。**

这句话更适合个人和小团队用户。

## 11. MVP 应该怎么收

### 商业化 MVP

第一版真正能收费的最小集合，我建议控制在这些能力：

- 云端注册和登录
- 创建工作区
- 创建和运行云端 Agent
- 任务创建、分配、状态追踪
- 基础 Skill 模板
- 任务历史记录
- 订阅和额度页面

### Version 1.5

- 团队邀请
- 更强的任务搜索
- 更好的任务模板
- 失败重试和结果复盘
- 更细的使用量展示

### Version 2

- 模板市场
- 更强的协作通知
- 任务自动化链路
- 团队级分析报表

## 12. 关键指标

这个产品更应该看产品增长指标，而不是企业销售指标：

- 注册到首个任务完成的转化率
- 试用到付费转化率
- Pro 套餐转化率
- Team 套餐转化率
- 周活跃用户数
- 每用户每周完成任务数
- 次月留存
- 额度增购率

### 北极星指标

建议用：

**每个活跃用户每周完成的 Agent 任务数**

这个指标最能反映产品是不是被持续使用。

## 13. 90 天执行计划

### 第 1-30 天

- 明确品牌、定位和首页文案
- 跑通云端基础流程
- 上线注册、价格页、订阅页
- 补基础埋点
- 做一版简洁的新手引导

### 第 31-60 天

- 做额度和计费
- 做 Starter / Pro / Team 套餐
- 打磨核心任务流程
- 邀请第一批个人开发者和小团队试用
- 连续发布演示内容

### 第 61-90 天

- 根据试用反馈优化核心体验
- 上线模板和最佳实践内容
- 强化留存和转化路径
- 开始做内容矩阵和社区传播

## 14. 风险与应对

### 风险 1：个人用户付费意愿不稳定

应对：

- 先让用户快速完成第一个有效任务
- 套餐边界清楚
- 强化“省时间”的价值感

### 风险 2：云端执行成本过高

应对：

- 严格控制试用额度
- 做并发、时长、次数三类限额
- 把高消耗能力放到付费层

### 风险 3：用户觉得和 IDE Agent 差别不大

应对：

- 强调持续任务推进能力
- 强调多任务管理能力
- 强调小团队协作能力

### 风险 4：产品太重，小用户不愿学习

应对：

- 首页和产品都要更轻
- 减少复杂配置项
- 默认提供模板和预设流程

## 15. 最终建议

如果你明确要走：

- 只提供云端使用
- 只做个人和小团队

那这套产品就应该从“企业 Agent 管理平台”转成：

**轻量、可自助、可快速上手的云端 AI Agent 工作台。**

它的商业重点不再是治理和部署，而是：

- 上手快
- 任务推进顺
- 价格门槛低
- 小团队协作自然
